■ 문서 운칙 (필독)
--------------------------------------------------------------------------------
• 최종본 본문([1] 이하)을 수정하거나, 「최종본에서는 이렇다」는 합의·결정이 생기면
  이 파일 하단 「변경 기록」에 반드시 날짜를 붙여 한 줄 이상 추가한다.
• 날짜 형식: YYYY-MM-DD (한국시각 KST 기준을 권장).
• 형식 예:
    YYYY-MM-DD  —  무엇을 바꿨는지·무엇이 합의됐는지 한 줄 요약 (필요 시 [N] 절과 대응)
• 본문 수정과 변경 기록 항목은 같은 작업에서 함께 반영한다.
• 변경 기록은 시간순으로 아래에 이어 붙인다(맨 아래가 가장 최근).


[1] BroG와 MyG의 역할 구분 (핵심)
--------------------------------------------------------------------------------
• BroG
  - 공개 맛집 정보. 서버(PostgreSQL 등)가 진실의 원천(Source of Truth).
  - 지도·목록·상세·등록·관리는 서버 API와 연동.
  - 배포 시 백엔드·DB·(필요 시) 정적 업로드 스토리지가 전제.

• MyG
  - 개인용 기록(메모·사진·제보 성격). BroG와 동일한 “한 줄의 공개 레코드”가 아님.
  - 모바일 앱(Capacitor 등) 구상 시: MyG는 폰 내부 SQLite(또는 동급 로컬 저장)에 두는 방향.
  - BroG와 “양방향 동기화”하지 않음. 데이터 종류·저장 위치가 처음부터 다름.

• 관계 한 줄 요약
  BroG = 서버 공개 DB / MyG = (웹은 현재 서버 MyG DB, 앱은) 로컬 개인 저장 + BroG는 읽기용으로만 갱신.


[2] BroG를 폰에서 보는 방식
--------------------------------------------------------------------------------
• BroG는 폰에서 “항상 온라인일 때만” 본다 (1차).
• 오프라인 캐시로 BroG를 보는 방식은 당분간 채택하지 않음.
• MyG가 폰에 있으므로 네트워크 끊김 시 불편 완화는 MyG 쪽에서 담당.


[3] BroG → MyG (가져오기)
--------------------------------------------------------------------------------
• 웹(broke)에는 이미 구현됨: BroG 상세·관리 화면의 「MyG로 내려받기」
  → API: POST /known-restaurants/posts/from-brog/{restaurantId}
• 모바일·로컬 MyG 설계 시 방향(합의):
  - 텍스트 + 이미지 URL(링크) 중심으로 정리. 파일을 폰에 미러링하지 않아도 됨.
• 현재 서버 구현은 BroG 이미지를 /uploads/myg/ 로 복사하는 경우가 있음 — 웹 서버 MyG와
  “폰 전용 로컬 MyG”는 구현 단계에서 구분해 맞출 것.


[4] MyG → BroG (올리기)
--------------------------------------------------------------------------------
• 공개 BroG로의 “무분별 자동 반영”은 검수·중복·스팸 이슈로 위험할 수 있음 →
  규칙·검증·(선택) 승인 등 설계가 필요하다는 전제는 유지.
• UX 합의 (최종에 가까운 결정):
  - 한 번에 자동 등록까지 시도해도 됨.
  - 사용자에게 데이터 파이프라인(매핑 단계)을 설명할 필요는 없음.
  - 실패 시: 상세 원인 나열 대신, 빠른 탈출구로
    「BroG 등록(작성) 화면에서 직접 작성해 주세요」류 안내 + 해당 페이지로 유도.
  - 운영/디버깅용으로 서버·로그에는 실패 원인을 남기는 것은 별도(사용자 노출과 분리).


[5] 웹 vs 앱(향후) 정리
--------------------------------------------------------------------------------
• 지금 broke+gourmet 웹: MyG는 서버 DB + MyG 전용 업로드 경로(/uploads/myg/).
• 향후 앱: BroG는 온라인만; MyG는 로컬; BroG 목록은 API로 받아 화면만 갱신(서버 SoT 유지).
• 안드로이드에 백엔드+PostgreSQL+프론트를 통째로 넣어 “폰 localhost만”으로 BroG 운영하는 방식은
  비현실적에 가깝다는 합의 — 검증은 PC/스테이징 서버 + 폰 클라이언트 조합이 현실적.


[6] 기타 (이번 스레드에서 맞춘 UI·용어)
--------------------------------------------------------------------------------
• 구 필터 “전체” 라벨: BroG·MyG 공통으로 「전체 보기」로 통일.
  예전 URL 값(전체보기, all 등)은 클라이언트에서 정규화.
• BroG → MyG 가져오기 버튼 문구(웹): 「MyG로 내려받기」.


[7] 문서 목적
--------------------------------------------------------------------------------
• 이 파일은 “최종본” 제품 방향을 한 곳에 묶어 두기 위한 요약이며,
  구현 세부(API 필드명·DB 스키마)는 코드·OpenAPI·sql/ 주석을 따름.
• 누적 변경·합의는 반드시 아래 「변경 기록」에 날짜를 붙여 남긴다(위 문서 운칙).
